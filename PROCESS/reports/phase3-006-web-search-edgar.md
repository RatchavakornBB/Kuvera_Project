## Result: ✅ DoD met — Phase 3 (Contracts, Concierge, Chat) complete

Gate: real end-to-end WebSocket tests, no mocks, twice (before and after a bug fix). `web_search_20250305` is Anthropic's official server tool — passing it in `tools=[...]` is sufficient; Claude executes the search server-side and returns real citations (`url`, `title`, `cited_text`) embedded in the response, no client-side tool-use loop needed. SEC EDGAR fetcher does the real two-step call (`company_tickers.json` CIK lookup, then `data.sec.gov/submissions/CIK{...}.json` filing fetch) against the live SEC API with a compliant User-Agent, no key required — verified directly against real NVDA and Medtronic data before wiring it into the node.

**Bug found and fixed via the end-to-end test** (not hypothetical): asking "Look up Medtronic's most recent 10-K filing on SEC EDGAR" initially returned filings for **Outlook Therapeutics**, not Medtronic. Root cause: the hard-routed company-name extraction heuristic pulled "Look" (the sentence's leading capitalized word) as its first candidate, and `lookup_cik`'s substring match (`needle in title.lower()`) matched "look" inside "Outlook Therapeutics." Notably, Claude itself caught the inconsistency mid-answer ("the filing list you provided is actually for Outlook Therapeutics, not Medtronic") and self-corrected using web_search to find the right company — the final answer was still correct, but the underlying EDGAR-routing bug was real and worth fixing, not something to leave relying on the model to paper over. Fixed with two changes: (1) a stoplist of imperative/question-starter words (look, please, show, what, find, ...) filtered out of extraction candidates, (2) `lookup_cik`'s match changed from a bare substring check to a word-boundary regex. Re-ran the identical failing query — now correctly matches Medtronic plc on the first try, 8 correct citations.

Deviations from spec: none load-bearing.

Risks: the EDGAR company-name extraction is still a light heuristic (capitalized-word-sequence regex + a stoplist), not a real NER model — it will still miscue on sufficiently unusual phrasing. Acceptable for this week's demo scope (the explicit "look up X's 10-K" pattern it's designed for works correctly now), but not bulletproof against arbitrary phrasing.
